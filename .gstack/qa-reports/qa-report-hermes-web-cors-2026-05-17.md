## Root Cause

**Bug:** `internal/auth/middleware.go` intercepted `OPTIONS` preflight requests before they reached `proxy.Handler()`.

**Impact:** When Hermes Web was accessed through Portal proxy (`/proxy/8787/`), any CORS preflight (`OPTIONS`) request was intercepted by the auth middleware, which returned `302 Found → /login`. The browser received the redirect instead of CORS headers, and blocked the actual request. Chat input → API call → preflight failure → CORS error.

---

## Fix

### Fix 1: Auth middleware allow OPTIONS preflight

**File:** `internal/auth/middleware.go:16`

```go
// Before:
if strings.HasPrefix(r.URL.Path, "/login") ||
    strings.HasPrefix(r.URL.Path, "/csrf") ||
    r.URL.Path == "/api/health" {

// After:
if r.Method == http.MethodOptions ||
    strings.HasPrefix(r.URL.Path, "/login") ||
    strings.HasPrefix(r.URL.Path, "/csrf") ||
    r.URL.Path == "/api/health" {
```

**Why:** CORS preflight (`OPTIONS`) requests must never be intercepted by auth middleware. They carry no credentials and should always pass through to the proxy handler, which returns `204 No Content` with proper `Access-Control-Allow-*` headers.

## Verification

### Before Fix
```
curl -v -X OPTIONS http://localhost:8747/proxy/8787/api/chat/start
  → HTTP/1.1 302 Found  (Location: /login)
  → No CORS headers
```

### After Fix
```

  -H "Origin: http://localhost:8747" \
  -H "Access-Control-Request-Method: POST"

  → HTTP/1.1 204 No Content
  → Access-Control-Allow-Credentials: true
  → Access-Control-Allow-Headers: Content-Type
  → Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS, HEAD, PATCH
  → Access-Control-Allow-Origin: http://localhost:8747
  → Access-Control-Max-Age: 3600
```

## Other Findings
